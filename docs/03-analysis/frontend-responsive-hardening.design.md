# Frontend Responsive Hardening Design

## 목적

도쿄 가족여행 페이지를 `desktop`, `Android`, `iPhone`, `iPad` 환경에서 안정적으로 읽고 조작할 수 있도록 UI 구조를 보강한다.  
이번 설계의 핵심은 “예쁘게 보이는 반응형”이 아니라, `sticky 레이아웃`, `터치/키보드 조작`, `지도 해석성`, `모바일 정보 밀도`가 플랫폼 차이에도 무너지지 않게 만드는 것이다.

## 문제 정의

현재 UI는 시각적 방향성은 좋지만, 다음과 같은 구조적 취약점이 있다.

1. `content-tabs`와 `section-jumpbar`가 고정 숫자 기반 `top` 값에 의존한다.
2. 예산/날씨/지도 날짜 스위치 등 2차 조작 버튼은 포커스 가시성이 약하다.
3. 지도에서 `메인 동선`과 `선택형 보조 포인트`가 코드상으로는 분리되어 있지만, 화면에서는 그 차이가 즉시 이해되지 않는다.
4. 모바일과 태블릿에서 카드 내 `meta`, `링크`, `액션 버튼`, `support-box`가 동시에 강하게 보여 정보 위계가 평평해진다.

이 문제들은 특히 `iPhone Safari`, `iPad Safari`, `Android Chrome`에서 겹침, 오조작, 해석 혼란으로 이어질 수 있다.

## 설계 목표

이번 보강은 아래 5가지를 만족해야 한다.

1. `sticky` 요소가 실제 렌더 높이에 따라 정렬되고 safe-area와 함께 동작할 것
2. 주요 인터랙션 컨트롤이 터치와 키보드 모두에서 명확하게 조작 가능할 것
3. 지도 범례가 메인 동선과 선택형 포인트를 즉시 설명할 것
4. 모바일/태블릿에서 카드 정보가 읽기 중심으로 정리될 것
5. 기존 비주얼 톤과 콘텐츠 구조를 유지하면서 최소 변경으로 해결할 것

## 비목표

이번 작업에서 하지 않는 것:

- 페이지 전체 리디자인
- 정보 구조 자체의 대규모 재편
- 프레임워크 전환 또는 라이브러리 교체
- Leaflet 지도 기능 확장
- JSON 콘텐츠 대량 재작성

즉, 이번 작업은 `구조 안정화`와 `조작성 개선`에 집중한다.

## 대상 파일과 책임

### `docs/assets/js/app.js`

책임:

- 실제 렌더된 `sticky` UI 높이 계산
- 계산 결과를 CSS 변수로 전달
- 지도 범례 렌더링
- 탭/점프바/리사이즈 후 레이아웃 기준값 재동기화

원칙:

- 기존 데이터 렌더링 흐름은 유지한다.
- 레이아웃 보정용 로직만 추가한다.
- 콘텐츠 의미를 바꾸는 수정은 하지 않는다.

### `docs/assets/css/styles.css`

책임:

- 하드코딩된 sticky 위치를 CSS 변수 기반으로 전환
- 포커스 스타일 일관화
- 터치 타깃 최소 크기 보장
- 지도 범례 스타일 정의
- 모바일/태블릿에서 정보 위계와 밀도 조정

원칙:

- 현재 색감, 타이포, 카드 질감은 유지한다.
- 같은 역할의 인터랙션은 같은 포커스 규칙을 쓴다.
- 특정 기기용 예외를 늘리기보다 공통 규칙으로 문제를 줄인다.

### `docs/index.html`

책임:

- 지도 범례용 마운트 포인트 제공

원칙:

- 구조 변경은 최소화한다.
- JS가 붙일 수 있는 안정적인 DOM 지점만 추가한다.

## 핵심 설계 결정

### 1. Sticky 레이아웃은 “고정 숫자”가 아니라 “실측값”으로 맞춘다

기존 문제는 `section-jumpbar`가 `74px`, `70px` 같은 숫자에 기대고 있다는 점이다.  
이 방식은 탭 줄 수, 폰트 확대, iOS safe-area, 번역 길이 변화에 약하다.

대신 다음 구조로 바꾼다.

- JS가 `.content-tabs`와 `.section-jumpbar`의 실제 높이를 측정
- `--sticky-tabs-height`
- `--sticky-jumpbar-height`
- `--sticky-stack-offset`
  를 루트 CSS 변수로 설정
- CSS는 이 변수만 사용해 `top`과 `scroll-padding-top`을 계산

이렇게 하면 화면 폭, 글자 크기, 플랫폼 차이가 있어도 sticky 정렬이 자연스럽게 유지된다.

추가로, 실측값 동기화는 단순 `resize`만으로는 부족하다. 다음 시점은 설계상 필수 트리거로 본다.

- 초기 렌더 완료 직후
- `setContentTab()` 이후
- `renderSectionJumpbar()` 이후
- `document.fonts?.ready` 완료 이후 1회
- `resize` / `orientationchange` 이후

특히 웹폰트 로드 후 높이가 바뀌는 경우가 있어, `font load 이후 재측정`은 iPhone/iPad에서 중요하다.

### 2. 조작성은 1차 버튼뿐 아니라 2차 컨트롤까지 동일한 수준으로 보장한다

현재 상단 탭은 포커스가 보이지만, 예산 스위치/날씨 스위치/지도 날짜 버튼은 상대적으로 약하다.  
여행 페이지 특성상 사용자는 정보 소비뿐 아니라 빈번한 상태 전환을 수행하므로, 이 버튼들은 사실상 핵심 내비게이션이다.

따라서:

- `content-tabs`
- `section-jumpbar`
- `budget-switch`
- `weather-switch`
- `map-toolbar`
- `action-chip`
- `timeline-link`
- `map-link`

전체에 일관된 `:focus-visible` 스타일을 적용한다.

또한 모바일 기준으로 최소 터치 높이 `44px`를 보장한다.

### 3. 지도는 “보기 좋은 지도”보다 “오해하지 않는 지도”가 우선이다

현재 코드상으로는 메인 동선과 선택형 포인트가 분리되어 있지만, 사용자는 시각적 설명 없이는 그 차이를 즉시 이해하기 어렵다.

이를 해결하기 위해 지도 상단에 작은 범례를 둔다.

- 메인 동선: 메인 루트 점/선
- 선택형 보조 포인트: 별도 점

범례는 설명 문단이 아니라 즉시 읽히는 칩 형태로 제공한다.  
이 선택은 모바일에서도 공간을 적게 차지하면서 해석 속도를 높인다.

또한 범례는 색상만으로 구분하지 않는다. 다음 중 하나 이상을 함께 제공해야 한다.

- 메인 동선은 `선 샘플`, 선택형은 `점 샘플`
- 또는 `메인`, `선택` 텍스트 라벨을 시각 요소와 함께 표기

이 원칙은 색각 차이, 화면 밝기 저하, 야외 사용 환경에서도 해석성을 유지하기 위한 최소 조건이다.

### 4. 모바일 정보 밀도는 “삭제”가 아니라 “위계 조정”으로 해결한다

이번 페이지는 정보량 자체가 가치이므로, 모바일에서 단순히 내용을 줄이는 것은 바람직하지 않다.  
대신 다음 원칙으로 조정한다.

- `meta`는 더 가볍게
- `핵심 지원 정보`는 더 선명하게
- 링크와 액션은 한 덩어리로 묶어 보이게
- 작은 화면에서는 여백과 그룹 단위를 더 분명히

즉, 내용은 유지하되 읽는 순서를 더 명확히 만든다.

## 플랫폼별 기대 동작

### Desktop

- 탭과 점프바가 겹치지 않는다.
- 키보드 탐색 시 포커스가 명확하다.
- 지도 범례가 루트/선택 포인트 차이를 바로 설명한다.

### Android Chrome Portrait

- 상단 탭이 가로 스크롤되어도 페이지 전체가 흔들리지 않는다.
- 버튼/칩/링크를 오조작 없이 누를 수 있다.
- 긴 한글/영문 혼합 텍스트가 카드 밖으로 튀어나오지 않는다.

### iPhone Safari Portrait

- safe-area를 침범하지 않는다.
- sticky stack이 상단 노치 환경에서도 안정적이다.
- 점프바와 탭이 서로 겹치지 않는다.
- 작은 폭에서도 버튼과 링크가 눌릴 만한 크기를 유지한다.

### iPad Safari Portrait / Landscape

- 태블릿에서 “모바일 압축 UI”처럼 보이지 않는다.
- sticky 요소가 landscape에서 겹치지 않는다.
- 지도와 카드가 동시에 답답하지 않다.

## 검증 전략

설계 완료 후 구현 검증은 다음 순서로 한다.

1. `node --check docs/assets/js/app.js`
2. sticky 변수/범례/포커스 셀렉터 존재 여부 구조 검증
3. `docs/responsive-qa-checklist.md` 기준 수동 QA
4. 남는 항목은 `NEED REAL DEVICE`로 분리 기록

이번 작업의 완료 조건은 단순히 CSS가 적용되는 것이 아니라:

- 구조 검증 통과
- 주요 상호작용 포인트가 키보드/터치에서 식별 가능
- sticky 겹침 리스크 제거
- 지도 해석성 개선

까지 포함한다.

검증 결과는 다음 두 등급으로 나눈다.

- `Blocking`
  - sticky 겹침
  - 가로 스크롤 발생
  - 버튼/링크/칩 터치 불가
  - 메인 동선과 선택형 포인트 오해 유발
- `Non-blocking / NEED REAL DEVICE`
  - Safari 고유 스크롤 감각 차이
  - 폰트 렌더 차이
  - hover/active 미세 차이

즉, `Blocking`이 하나라도 남아 있으면 구현 완료로 보지 않는다.

## 리스크와 대응

### 리스크 1. JS 측정 시점이 너무 이르면 값이 0이 될 수 있음

대응:

- 초기 렌더 후 `requestAnimationFrame` 기반 동기화
- `resize` / `orientationchange` 재동기화
- 탭/점프바 콘텐츠 갱신 뒤에도 다시 호출

### 리스크 2. CSS 변수 도입 후 특정 breakpoint에서 이전보다 간격이 커질 수 있음

대응:

- 변수 기반 계산을 쓰되, 여백 상수는 최소로 유지
- 작은 화면에서는 상수 값을 더 줄여 과도한 빈 공간을 막음

### 리스크 3. 인터랙션 강조가 많아져 시각적으로 번잡해질 수 있음

대응:

- 포커스는 키보드 사용자에게만 보이는 `:focus-visible` 사용
- hover와 active를 같은 강도로 올리지 않음

## 성공 기준

다음이 만족되면 이 설계는 성공이다.

1. `sticky` 구조가 iOS/Android/desktop에서 겹침 없이 작동한다.
2. 예산/날씨/지도/체크 버튼이 키보드와 터치 모두에서 명확하게 조작된다.
3. 지도에 대해 “이 점도 다 가는 코스인가?”라는 혼동이 줄어든다.
4. 모바일과 태블릿에서 카드가 덜 답답하고, 읽는 순서가 더 자연스럽다.
5. 기존 여행 페이지의 분위기와 정보량은 유지된다.
